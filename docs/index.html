Natural language
for any robot.

ros2_lingua is a ROS 2 library that translates natural language instructions into validated, dependency-aware execution plans — dispatched over your existing ROS 2 actions and services. 

# Send a natural language instruction
ros2 run ros2_lingua_mock cli "go to the table and pick up the bottle"

# The library figures out the rest:
✅  Plan generated — 3 step(s):
   1.  stabilize_robot        ← auto-chained prerequisite
   2.  navigate_to_location   params: {location_name: "table"}
   3.  pick_up_object         params: {object_name: "bottle"}
Designed for versatility. ros2_lingua works with any ROS 2 robot — wheeled AMRs, robotic arms, humanoids, drones, AUVs. Your nodes describe what they can do. The library handles the rest.

Quick Start

Get ros2_lingua running in under 10 minutes. You need ROS 2 Humble and Ollama installed.

1

mkdir -p ~/ros2_lingua_ws/src && cd ~/ros2_lingua_ws/src
git clone https://github.com/YOUR_USERNAME/ros2_lingua.git
cp -r ros2_lingua/ros2_lingua_core .
cp -r ros2_lingua/ros2_lingua .
cp -r ros2_lingua/ros2_lingua_mock .

cd ~/ros2_lingua_ws
source /opt/ros/humble/setup.bash
colcon build && source install/setup.bash
2

Install rosbridge

sudo apt install ros-humble-rosbridge-suite
3

Launch the demo system

Open two terminals. In the first:

# Terminal 1 — rosbridge WebSocket server
ros2 launch rosbridge_server rosbridge_websocket_launch.xml

In the second:

# Terminal 2 — full demo system
ros2 launch ros2_lingua_mock demo.launch.py
4

Send an instruction

# In a third terminal
ros2 run ros2_lingua_mock cli "go to the table and pick up the bottle"

Open http://localhost:8080 to see the mission control dashboard.

Core Concepts

ros2_lingua is built on three ideas. Understanding them makes everything else obvious.

Capability Schema
Every robot node declares what it can do in a structured schema. This is the single source of truth the LLM reasons from.
Grounding Engine
Translates natural language into a validated plan using only registered capabilities. Hallucinated capabilities are caught before execution.
Backward Chaining
Automatically resolves prerequisites. If navigate requires balance and balance isn't achieved, stabilize is inserted automatically.
State Tokens
Simple string tokens like "robot_is_balanced" track what's currently true. Nodes set and clear tokens as they execute.

Capabilities

A Capability is the fundamental unit of ros2_lingua. It describes one thing a node can do in a way that both the LLM and the planner can understand.

from ros2_lingua_core import Capability, CapabilityParameter, Tags

navigate = Capability(
    # Identity — used in plans and LLM prompts
    name="navigate_to_location",
    description="Moves the robot to a named location in the environment",

    # ROS 2 interface — exactly one of these
    ros_action="robot/navigate_to_pose",  # or ros_service=

    # Input parameters
    parameters=[
        CapabilityParameter(
            name="location_name",
            type="string",
            description="Where to go, e.g. 'table', 'door', 'charging_dock'",
            required=True,
        ),
        CapabilityParameter(
            name="speed",
            type="float",
            description="Speed as fraction of maximum (0.1–1.0)",
            required=False,
            default=0.5,
        ),
    ],

    # Preconditions: must be true before this can run
    preconditions=["robot_is_balanced"],

    # Postconditions: become true after this runs
    postconditions=["robot_at_location"],

    # Tags: for filtering and categorization
    tags=[Tags.LOCOMOTION, Tags.NAVIGATION],
)
Write descriptions for the LLM, not for humans. The description and parameter description fields are injected directly into the LLM prompt. The more specific and clear they are, the better the grounding quality will be. Mention example values. Mention constraints.

CapabilityParameter types

The type field is a string used in the LLM prompt. Use Python/JSON primitive names for simple types, and ROS message type strings for complex ones:

Type stringMeaning
"string"Text value
"float"Decimal number
"int"Integer number
"bool"True or false
"geometry_msgs/Pose"ROS Pose message
any stringPassed as-is to the LLM for context

Capability Registry

The CapabilityRegistry stores all registered capabilities and the robot's current symbolic state. It's the single source of truth the grounding engine reads from.

from ros2_lingua_core import CapabilityRegistry

registry = CapabilityRegistry()

# Register capabilities
registry.register(navigate)      # raises if already registered
registry.update(navigate)        # register or overwrite
registry.unregister("navigate_to_location")

# Query capabilities
registry.get("navigate_to_location")         # by name
registry.get_all()                               # all capabilities
registry.get_by_tag("locomotion")             # by single tag
registry.get_by_tags(["locomotion", "manipulation"], match="any")
registry.get_by_tags(["manipulation", "social"], match="all")
registry.get_all_tags()                          # sorted unique tag list
registry.get_untagged()                          # caps with no tags

# State management
registry.set_state("robot_is_balanced")
registry.clear_state("robot_is_balanced")
registry.get_state()                             # current state set
registry.is_satisfied(["robot_is_balanced"])  # check preconditions

Grounding Engine

The GroundingEngine translates natural language into ActionPlan objects using the LLM backend and the capability registry.

from ros2_lingua_core import GroundingEngine, OllamaBackend

engine = GroundingEngine(
    registry=registry,
    backend=OllamaBackend(model="llama3.1"),
    auto_chain=True,    # insert missing prerequisites automatically
    max_retries=3,       # retry on LLM timeout/rate limit
    retry_delay=1.0,     # initial wait between retries (seconds)
    retry_backoff=2.0,   # multiply wait by this on each retry
)

# Ground an instruction
plan = engine.ground("go to the table and pick up the bottle")

# Ground with tag filter — only consider locomotion capabilities
plan = engine.ground("move to position A", tag_filter=["locomotion"])

# Check the result
if plan.feasible:
    for step in plan.steps:
        print(step.capability_name, step.parameters)
else:
    print("Not feasible:", plan.reason)

Backward Chaining

The backward chaining planner automatically resolves capability prerequisites without you hardcoding execution sequences. It works by tracing the dependency graph backward from the goal.

# Given these capabilities:
# stabilize_robot  → produces: [robot_is_balanced]
# navigate         → requires: [robot_is_balanced]
#                    produces: [robot_at_location]
# pick_up_object   → requires: [robot_at_location, arm_is_free]

# Current state: {arm_is_free}  (robot_is_balanced not yet true)

chain = registry.resolve_chain(
    "pick_up_object",
    current_state={"arm_is_free"}
)

# Result: [stabilize_robot, navigate, pick_up_object]
# stabilize was inserted because navigate needed robot_is_balanced
Design your state tokens carefully. The quality of automatic chaining depends on your precondition/postcondition design. Use specific, unambiguous token names. Prefer "robot_is_balanced" over "ready". Each token should have exactly one capability that produces it.

Tags

Tags let you categorize capabilities and filter what the LLM sees when grounding an instruction. For robots with large capability sets, filtering to relevant tags significantly improves grounding quality.

from ros2_lingua_core import Tags

# Standard tag constants
Tags.LOCOMOTION    # "locomotion"  — movement, navigation, driving
Tags.MANIPULATION  # "manipulation" — arms, grippers, pick/place
Tags.BALANCE       # "balance"      — stabilization, posture
Tags.SPEECH        # "speech"       — TTS, STT, voice I/O
Tags.SOCIAL        # "social"       — gestures, HRI
Tags.PERCEPTION    # "perception"   — cameras, lidar, detection
Tags.MAPPING       # "mapping"      — SLAM, localization
Tags.SYSTEM        # "system"       — power, modes, diagnostics
Tags.SAFETY        # "safety"       — e-stop, collision avoidance
Tags.INSPECTION    # "inspection"   — for ROVs, drones, industrial

# Custom tags — any string works
tags=["outdoor", "requires_gps"]

# Filter grounding to specific tags
plan = engine.ground("go to waypoint A", tag_filter=[Tags.LOCOMOTION])
Untagged capabilities are always included. When using a tag filter, capabilities with no tags are always shown to the LLM. This prevents accidentally hiding capabilities that haven't been tagged yet.

Integrating With Your Robot

Adding ros2_lingua to an existing robot takes four steps. Each step is independent — you can add them incrementally as you integrate.

Step 1 — Declare Capabilities

For each action or service your robot exposes, write a Capability object. This is the only part that requires knowledge of your specific robot.

# my_robot/capabilities.py
from ros2_lingua_core import Capability, CapabilityParameter, Tags

# One capability per action/service your robot exposes
NAVIGATE = Capability(
    name="navigate_to_location",
    description="Drives the robot to a named location. "
                "Known locations: kitchen, office, dock, hallway.",
    ros_action="my_robot/navigate",
    parameters=[
        CapabilityParameter("location_name", "string", "Destination name"),
    ],
    preconditions=["robot_is_ready"],
    postconditions=["robot_at_location"],
    tags=[Tags.LOCOMOTION],
)

PICK = Capability(
    name="pick_up_object",
    description="Picks up a named object with the robot arm.",
    ros_action="my_robot/pick",
    parameters=[
        CapabilityParameter("object_name", "string", "Object to pick up"),
    ],
    preconditions=["robot_at_location", "arm_is_free"],
    postconditions=["object_in_hand"],
    tags=[Tags.MANIPULATION],
)

Step 2 — Add LinguaMixin

Inherit from LinguaMixin alongside Node and call register_lingua_capability() in your constructor. That's the entire integration for the registration side.

from rclpy.node import Node
from ros2_lingua import LinguaMixin
from my_robot.capabilities import NAVIGATE

class MyNavigationNode(LinguaMixin, Node):
    def __init__(self):
        Node.__init__(self, "my_navigation_node")
        LinguaMixin.__init__(self)

        # ... your existing __init__ code ...
        self._action_server = ActionServer(...)

        # Register with ros2_lingua
        self.register_lingua_capability(NAVIGATE)
Inheritance order matters. Always put LinguaMixin before Node in the inheritance list, and call both __init__ methods explicitly. Python's MRO requires this ordering.

Step 3 — Subclass the Dispatcher

The dispatcher node executes plans by calling your robot's actual actions and services. Since every robot has different message types, you subclass DispatcherNode and override the execution methods.

from rclpy.action import ActionClient
from ros2_lingua.dispatcher_node import DispatcherNode

# Import your robot's action/service types
from my_robot_interfaces.action import NavigateTo
from my_robot_interfaces.action import PickObject

class MyRobotDispatcher(DispatcherNode):

    def _call_action(self, action_name: str, cap_name: str, params: dict) -> bool:

        if cap_name == "navigate_to_location":
            client = ActionClient(self, NavigateTo, action_name)
            client.wait_for_server(timeout_sec=5.0)

            goal = NavigateTo.Goal()
            goal.location_name = params.get("location_name", "")
            goal.speed = params.get("speed", 0.5)

            future = client.send_goal_async(goal)
            # ... wait for result ...
            return True

        if cap_name == "pick_up_object":
            # ... similar pattern ...
            return True

        # Unknown capabilities fall back to demo mode
        return super()._call_action(action_name, cap_name, params)

Step 4 — Report State Changes

Call update_lingua_state() whenever something significant happens — after an action completes, when a sensor detects a condition, when a mode changes. This is what enables automatic prerequisite chaining.

async def _execute_navigate(self, goal_handle):
    # ... execute your navigation ...
    result = await self._do_navigation(goal_handle.request)

    if result.success:
        # Tell lingua the state has changed
        self.update_lingua_state(
            set_tokens=["robot_at_location"],
        )
        goal_handle.succeed()
    else:
        # Clear the token if navigation failed
        self.update_lingua_state(
            clear_tokens=["robot_at_location"],
        )
        goal_handle.abort()
Set initial state at node startup. Call update_lingua_state() in __init__ to set any tokens that are true when your node boots — for example set_tokens=["arm_is_free"] if the arm starts unloaded.

from ros2_lingua_core import OllamaBackend, OpenAIBackend, AnthropicBackend, RetryConfig

# Ollama — local, no API key, recommended for offline robots
backend = OllamaBackend(model="llama3.1", host="http://localhost:11434")

# OpenAI
backend = OpenAIBackend(api_key="sk-...", model="gpt-4o")

# Anthropic Claude

# Custom backend — implement one method
class MyBackend:
    def complete(self, messages: list[dict]) -> str:
        # call your LLM, return response string
        return my_llm_call(messages)

Error Handling

Every error in ros2_lingua is a specific exception type inheriting from LinguaError. Catch what you care about:

from ros2_lingua_core import (
    LinguaError,           # catch all ros2_lingua errors
    LLMBackendError,       # any LLM connectivity problem
    LLMTimeoutError,       # LLM took too long
    LLMRateLimitError,     # API rate limit hit
    LLMModelNotFoundError, # model doesn't exist
    HallucinationError,    # LLM referenced unknown capability
    UnsatisfiablePreconditionError, # planner couldn't satisfy prereq
    StepTimeoutError,      # step didn't complete in time
    StepFailedError,       # step returned failure
)

try:
    plan = engine.ground("pick up the bottle")
except LLMModelNotFoundError as e:
    print(f"Model not found: {e}. Try: ollama pull llama3.1")
except LLMBackendError as e:
    print(f"LLM connectivity issue: {e}")
Grounding never raises by default. The GroundingEngine.ground() method catches most errors and returns an infeasible ActionPlan with a descriptive reason string rather than raising. This keeps the grounding node stable under all conditions.

Architecture

ros2_lingua_core (pure Python, no ROS 2 dependencies) ├── schema.py Capability, CapabilityParameter, Tags ├── registry.py CapabilityRegistry, backward chain planner ├── grounding.py GroundingEngine, ActionPlan, ActionStep ├── backends.py OpenAI, Anthropic, Ollama, Mock, RetryConfig └── errors.py LinguaError hierarchy (11 specific types) ros2_lingua (ROS 2 interface layer) ├── grounding_node.py Service server wrapping GroundingEngine ├── dispatcher_node.py Executes ActionPlans over ROS 2 actions/services └── capability_mixin.py LinguaMixin for self-registering nodes ros2_lingua_interfaces (custom service/message definitions) ├── srv/RegisterCapability.srv ├── srv/GroundInstruction.srv ├── srv/UpdateState.srv └── msg/ExecutionStatus.msg ros2_lingua_mock (simulation for testing and demos) ├── balance_node.py Simulates balance controller ├── navigation_node.py Simulates locomotion ├── manipulation_node.py Simulates arm/gripper ├── speech_node.py Simulates TTS ├── robot_monitor.py Unified log stream ├── cli.py Command line instruction tool ├── dashboard_server.py HTTP server for web dashboard └── launch/demo.launch.py Starts everything with one command

Data flow

User instruction │ ▼ GroundingNode (/lingua/ground service) │ calls registry.to_llm_context(tags=...) │ calls LLM backend with capability context + instruction │ parses + validates LLM response │ runs backward chain planner (auto_chain=True) │ ▼ ActionPlan (validated, ordered steps) │ ├──▶ service response (plan_json) └──▶ /lingua/current_plan topic │ ▼ DispatcherNode │ for each step: │ look up capability │ call ros_action or ros_service │ wait for result │ update state via /lingua/update_state │ ▼ Your robot nodes
